> 版本: 1.0 | 日期: 2026-05-30
> 适用范围: Yaxiio 自身开发 + L5 生成的代码
> 基于 Martin Fowler、Google Readability、Uncle Bob 等业界共识

---

## 零、为什么需要代码质量宪章

代码质量的评价有极强的主观性——"这段代码灵活"和"这段代码很乱"，往往是同一个人在不同时间对同一段代码的评价。

Yaxiio 面临**两层代码质量挑战**：

| 层面 | 谁写的 | 问题 |
|------|--------|------|
| **Yaxiio 自身** | 人类开发者（我们） | 几百行工作流引擎、五层模块、零测试覆盖——代码腐化速度极快 |
| **L5 生成的代码** | AI Agent | 没有质量门禁的自动代码生成 = 熵增引擎 |

本宪章同时约束两层：**人类写的 Yaxiio 内核，和 AI 生成的执行代码，都过同一道门。**

## 一、执行机制

```
人类提交代码 ──→ CodeQualityScorer ──→ score ≥ 6 → 合入
                                         score < 6 → 打回 + 修复建议
                                         
L5 生成代码 ──→ CodeQualityScorer ──→ score ≥ 6 → 合入
                                         score < 4 → 暂停 + 通知人工
```

**关键规则**：

| 规则 | 适用对象 | 说明 |
|------|---------|------|
| **提交前自检** | 人类开发者 | `python3 tools/cq_check.py` 在 commit 前运行 |
| **CI 门禁** | 人类开发者 | 低于 6 分阻止合并 |
| **AI 生成门禁** | L5 产出的代码 | 低于 6 分自动打回给 Agent 修复 |
| **人工兜底** | L5 产出 < 4 分 | 暂停自动合并，通知人工 code review |
| **宪法级强制** | 全部 | 代码审查规则不可绕过 |

## 二、七大维度速查

| 维度 | 一句话 | 量化指标 | 权重 |
|------|--------|---------|------|
| **可维护性** | 修 bug 是否轻松 | 圈复杂度 + 耦合度 + 文件大小 | 20% |
| **可读性** | 同事能否读懂 | 命名规范 + 注释率 + 函数长度 | 20% |
| **可扩展性** | 加功能是否要改老代码 | 开闭原则 + 接口抽象度 | 15% |
| **灵活性** | 代码预留多少复用点 | 抽象层次 + 配置化程度 | 10% |
| **简洁性** | 是否最简单解法 | 冗余代码率 + 过度设计检测 | 15% |
| **可复用性** | 重复代码是否被抽取 | DRY 违规率 + 模块复用 | 10% |
| **可测试性** | 写单元测试是否容易 | 依赖注入率 + 纯函数比例 | 10% |

## 五、开发工作流集成

### 5.1 提交前自检命令

```bash
# 每次 commit 前运行
python3 tools/cq_check.py --changed        # 只检查变更文件
python3 tools/cq_check.py --all             # 全量检查
python3 tools/cq_check.py --file workflow_engine.py  # 指定文件

# 输出示例:
# ✅ constitution.py: 7.6/10 (maintainability=8, readability=8, ...)
# 🔴 workflow_engine.py: 4.0/10 — 3 violations:
#    - 圈复杂度过高 (CC=12, 函数 _orchestrate_subtasks, 阈值=10)
#    - 文件过大 (1231行, 阈值=500行)
#    - 嵌套过深 (5层, 阈值=4层)
```

### 5.2 CI 门禁配置

```yaml
# .github/workflows/code-quality.yml
- name: Code Quality Check
  run: |
    python3 tools/cq_check.py --changed --threshold 6
    # 低于 6 分的文件阻止合并
```
