# 雅溪 Yaxiio 设计哲学白皮书

> Version: 1.0 | Date: 2026-05-29
> 回答一个问题：为什么 Yaxiio 长这样？

---

## 一、不是又一个 Agent 框架

2024-2026 年涌现了大量 AI Agent 框架：LangChain、CrewAI、AutoGPT、Dify、Coze……它们解决的是同一个问题：**如何让 LLM 调用工具**。

Yaxiio 回答的是另一个问题：**一群 Agent 如何像一个团队一样自我管理、自我进化，而不需要人类在中间协调**。

这不是"Agent 协作"，这是"Agent 自治"。

---

## 二、为什么是五层而不是三层？

大多数框架的架构是三层：**感知 → 规划 → 执行**。看起来很自然。

Yaxiio 把三层拆成五层：**感知(L1) → 规划(L2) → 调度(L3) → 执行(L4) → 进化(L5)**。

### 2.1 为什么把"规划"拆成"规划+调度"？

**规划**回答"怎么做"：任务拆成几个子任务，每个子任务归谁。

**调度**回答"谁来做、什么时候做"：Agent 是否空闲、依赖是否满足、是否可以并行。

如果这两步不分开，就会出现一个常见问题：规划阶段分配了 Agent A，但 Agent A 正在忙，整个任务就卡住了。分开后，规划只需声明"这个子任务需要翻译官"，调度层根据实时状态决定用哪个翻译官实例——可能新建一个，可能等现有的空闲。

**实际代码体现**：`workflow_engine.py` 中 `_decompose_via_l2()` 只做拆解，`_orchestrate_subtasks()` 根据依赖关系和 Agent 就绪状态动态发射。

### 2.2 为什么要有独立的 L5 进化层？

每一轮任务结束后，L5 给结果打分。如果分数低，不是简单地"重试"——而是**分析为什么低**，然后改进。

- 是 prompt 不够好？→ 触发 prompt 优化
- 是知识不够？→ 触发 web 搜索补充
- 是 Agent 本身不适合？→ 换 Agent 类型重试

如果没有独立的 L5，这些改进逻辑就会分散在 L3/L4 中，变成一堆 if-else。独立出来之后，L5 可以独立迭代——换评分模型、换优化策略——不影响执行层。

**实际代码体现**：`workflow_engine._do_L5()` 返回 `verdict: "retry"/"reject"/"pass"`，`_analyze_gap()` 根据 verdict 决定下一步动作。

## 三、为什么 Agent 需要隔离沙箱而不是共享进程？

三个理由：

**安全**：Agent 执行的命令来自 LLM 输出。LLM 可能被诱导输出恶意命令。每个 Agent 跑在独立进程/Docker 容器里，一个炸了不影响其他。

**上下文隔离**：如果多个 Agent 共享一个进程，它们的 LLM 会话记忆会互相污染。审计官的"这个字段有问题"可能出现在翻译官的上下文中。

**资源控制**：独立进程意味着可以独立限制 CPU、内存、超时。翻译官慢了不会拖累审计官。

**实际代码体现**：`neuron.py` 是独立 Python 进程（`subprocess.Popen`），`sandbox_manager.py` 提供 Docker-in-Docker 容器级隔离。

## 四、为什么用四象限生命周期管理？

不是所有 Agent 都一样重要：

| 象限 | 含义 | 策略 | 示例 |
|------|------|------|------|
| **Core** | 核心业务，不可销毁 | 只重建，不销毁 | 翻译官 |
| **Strategic** | 按需创建，闲置回收 | 超时休眠 | 审计官 |
| **Utility** | 长期驻留 | 异常率高才重建 | 前端工程师 |
| **Ephemeral** | 用完即弃 | 任务完成即销毁 | 临时数据分析师 |

这个设计的核心思想是：**不是所有 Agent 都值得长期养着**。有些 Agent 是"临时工"，干完活就可以走。有些是"核心员工"，需要长期稳定运行。

如果所有 Agent 一视同仁，要么资源浪费（临时工长期占内存），要么频繁重建（核心员工反复重启）。

**实际代码体现**：`modules/layer2/lifecycle_manager.py` 定义了四象限映射规则。

## 五、为什么 Commander 只放在 L3？

Commander 是"流程的作者，不是执行者"。它的职责是：理解任务 → 设计工作流 → 调度 Agent → 等待结果。它不亲自执行任何业务逻辑。

如果 Commander 同时负责 L2（规划）和 L4（执行），它就会变成"既当裁判又当运动员"——自己规划的任务自己执行，失去了客观性。

放在 L3（调度层）是最合理的位置：它接收 L2 的规划，指挥 L4 执行，然后把结果交给 L5 评估。它是**信息的中转站**，不是**任务的执行者**。

**实际代码体现**：`yaxiio.py` 的 Commander 类只有 `handle_task()` → `constitution.review()` → `_run_delegated()`，没有 `_run_audit()`、`_run_translate()` 这类方法。

## 六、为什么用 PM2 双守护互保？

单层守护的经典问题：**谁守护守护者？**

```
Commander 挂了 → Guardian 重启 Commander ✅
Guardian 挂了 → ??? ❌
```

Yaxiio 的解法：

```
PM2 → Guardian-Primary + Guardian-Secondary
       │                    │
       ├─ 管理 Commander    ├─ 监控 Primary 心跳
       │                    │    Primary 失联 → 接管
```

PM2 是进程管理器（类似 systemd），只管 Guardian 进程本身——Guardian 崩了 PM2 重启它。Guardian 内部两个实例通过 Redis 竞选 Leader，只监控对方不监控自己。

**为什么不用 K8s？** 因为 Yaxiio 需要运行在单机上（甚至树莓派）。K8s 的假设是有多台机器和完整的集群基础设施。双守护是单机场景下的最小可行高可用方案。

**实际代码体现**：`entrypoint.sh` 使用 PM2 管理 Guard，`pi_guardian_v3.py` 的 `DualGuard` 类实现 Leader 选举。

## 七、什么是"上下文隔离 + 记忆继承"？

这是 Yaxiio 最容易被误解的设计。

**上下文隔离**：每任务新建 Agent 实例，LLM 对话上下文从零开始。

**记忆继承**：任务结束后，好的经验被写入 L0 经验库（Redis），下次同类任务从中检索参考。

这两个看似矛盾的概念，其实解决的是 AI Agent 的一个核心难题：

- 如果**不隔离**，上下文会无限膨胀，越用越慢、越用越乱（LangChain 的 Memory 问题）
- 如果**不继承**，每次都是"新人"，浪费历史经验

Yaxiio 的做法是"用完就忘，但把好东西存起来"——就像人类做完一个项目后写总结报告，下一个项目看总结报告而不是翻全部邮件。

**实际代码体现**：`neuron.py` 中 `MEMORY_KEY = f"agent:{AGENT_NAME}:{TASK_ID}:memory"` 实现隔离；`l0_memory.py` 中 `_save_experience()` 和 `_retrieve_experiences()` 实现继承。

## 八、设计原则速查

| # | 原则 | 含义 |
|---|------|------|
| 1 | **MCP-First** | 所有任务默认走五层流水线 |
| 2 | **纯编排** | Commander 只做路由和调度，不亲自执行 |
| 3 | **白名单准入** | 只有宪法明确授权可绕过流水线 |
| 4 | **LLM 决策** | 任务理解和分派由 LLM 驱动，禁止硬编码分支 |
| 5 | **沙箱隔离** | 代码执行必须在 L4 sandbox 内 |
| 6 | **审计不可绕过** | 所有操作记录在案 |
| 7 | **模克隆** | 每任务新 Agent 实例，上下文零泄露 |
| 8 | **渐进修复** | 故障先诊断再修复，不盲目重启 |

---

## 九、为什么这些选择重要？

AI Agent 领域正在从"单个 Agent 调用工具"走向"多 Agent 协作"。但大多数框架把"协作"等同于"多个 Agent 共享一个对话"。

Yaxiio 的核心主张是：**真正的 Agent 自治需要宪法约束、独立执行、自我进化**。这不是工程上的微调，而是范式的转变。

就像操作系统从单道批处理到多道分时系统的跃迁——不是让程序跑得更快，而是让多个程序**安全地共存、独立地运行、公平地调度**。

Yaxiio 想做的是 Agent 世界的操作系统内核。
