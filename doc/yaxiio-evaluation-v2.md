## 二、优化路线图

### 第一梯队：今天执行 / 1-2 小时 / 影响最大

#### 1. 代码目录统一
- **现状**: `/opt/commander/` + `/app/.pi/skills/` + `modules/` 三处分散
- **方案**: 核心代码收敛到 `/opt/commander/`，Guardian/Layers 保持 `/app/.pi/`，但通过软链统一入口
- **收益**: 新开发者 5 分钟找到入口（现在是 30 分钟）

#### 2. 能力卡片自恢复
- **现状**: 容器重建后 Redis 为空，需手动运行 upgrade_capability_cards.py
- **方案**: entrypoint.sh 启动时检查 `agent:card:*` 数量 < 12 → 自动从 `/app/.pi/cards/` JSON 文件导入
- **收益**: 消除容器重建后最常见的人工恢复步骤

#### 3. 健康检查增强
- **现状**: Guardian 只能 pgrep 查进程存活
- **方案**: Commander 加 `/health` HTTP 端点，返回内存、活跃Agent数、任务队列、L5评分趋势
- **收益**: 出问题 10 秒定位 vs 现在 10 分钟翻日志

### 第二梯队：本周执行 / 3-5 小时 / 质量质变

#### 4. 核心调度链路单测
- **范围**: `_execute_subtask` 的 Redis 直连调度 → READY → dispatch → respond
- **覆盖**: 正常流程 + READY 超时 + Agent 崩溃 + 结果超时
- **收益**: 80% 故障在提交前发现

#### 5. Docker Compose 生产化
- **内容**: compose.yml + healthcheck + restart policy + `/data` 持久化卷
- **收益**: 容器挂了 `docker compose up -d` 一键恢复

#### 6. 文档与代码同步
- **内容**: 从 capability-cards/_index.json 生成 Agent 表格
- **收益**: README 永远准确，不再手动维护

### 第三梯队：下个迭代

#### 7. 可观测性
- Redis 内存告警（>80% → 发事件）
- 任务延迟 P50/P95/P99
- Commander 重启趋势

#### 8. CI/CD
- GitHub Actions: push → 语法检查 → 单测 → 构建镜像
