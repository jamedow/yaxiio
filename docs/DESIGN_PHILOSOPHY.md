# 雅溪 Yaxiio 设计哲学白皮书

> Version: 1.0 | Date: 2026-05-29
> 回答一个问题：为什么 Yaxiio 长这样？

---

## 一、不是又一个 Agent 框架

2024-2026 年涌现了大量 AI Agent 框架：LangChain、CrewAI、AutoGPT、Dify、Coze……它们解决的是同一个问题：**如何让 LLM 调用工具**。

Yaxiio 回答的是另一个问题：**一群 Agent 如何像一个团队一样自我管理、自我进化，而不需要人类在中间协调**。

这不是"Agent 协作"，这是"Agent 自治"。

---

## 二、为什么是五层而不是三层？

大多数框架的架构是三层：**感知 → 规划 → 执行**。看起来很自然。

Yaxiio 把三层拆成五层：**感知(L1) → 规划(L2) → 调度(L3) → 执行(L4) → 进化(L5)**。

### 2.1 为什么把"规划"拆成"规划+调度"？

**规划**回答"怎么做"：任务拆成几个子任务，每个子任务归谁。

**调度**回答"谁来做、什么时候做"：Agent 是否空闲、依赖是否满足、是否可以并行。

如果这两步不分开，就会出现一个常见问题：规划阶段分配了 Agent A，但 Agent A 正在忙，整个任务就卡住了。分开后，规划只需声明"这个子任务需要翻译官"，调度层根据实时状态决定用哪个翻译官实例——可能新建一个，可能等现有的空闲。

**实际代码体现**：`workflow_engine.py` 中 `_decompose_via_l2()` 只做拆解，`_orchestrate_subtasks()` 根据依赖关系和 Agent 就绪状态动态发射。

### 2.2 为什么要有独立的 L5 进化层？

每一轮任务结束后，L5 给结果打分。如果分数低，不是简单地"重试"——而是**分析为什么低**，然后改进。

- 是 prompt 不够好？→ 触发 prompt 优化
- 是知识不够？→ 触发 web 搜索补充
- 是 Agent 本身不适合？→ 换 Agent 类型重试

如果没有独立的 L5，这些改进逻辑就会分散在 L3/L4 中，变成一堆 if-else。独立出来之后，L5 可以独立迭代——换评分模型、换优化策略——不影响执行层。

**实际代码体现**：`workflow_engine._do_L5()` 返回 `verdict: "retry"/"reject"/"pass"`，`_analyze_gap()` 根据 verdict 决定下一步动作。

## 三、为什么 Agent 需要隔离沙箱而不是共享进程？

三个理由：

**安全**：Agent 执行的命令来自 LLM 输出。LLM 可能被诱导输出恶意命令。每个 Agent 跑在独立进程/Docker 容器里，一个炸了不影响其他。

**上下文隔离**：如果多个 Agent 共享一个进程，它们的 LLM 会话记忆会互相污染。审计官的"这个字段有问题"可能出现在翻译官的上下文中。

**资源控制**：独立进程意味着可以独立限制 CPU、内存、超时。翻译官慢了不会拖累审计官。

**实际代码体现**：`neuron.py` 是独立 Python 进程（`subprocess.Popen`），`sandbox_manager.py` 提供 Docker-in-Docker 容器级隔离。

## 四、为什么用四象限生命周期管理？

不是所有 Agent 都一样重要：

| 象限 | 含义 | 策略 | 示例 |
|------|------|------|------|
| **Core** | 核心业务，不可销毁 | 只重建，不销毁 | 翻译官 |
| **Strategic** | 按需创建，闲置回收 | 超时休眠 | 审计官 |
| **Utility** | 长期驻留 | 异常率高才重建 | 前端工程师 |
| **Ephemeral** | 用完即弃 | 任务完成即销毁 | 临时数据分析师 |

这个设计的核心思想是：**不是所有 Agent 都值得长期养着**。有些 Agent 是"临时工"，干完活就可以走。有些是"核心员工"，需要长期稳定运行。

如果所有 Agent 一视同仁，要么资源浪费（临时工长期占内存），要么频繁重建（核心员工反复重启）。

**实际代码体现**：`modules/layer2/lifecycle_manager.py` 定义了四象限映射规则。

## 五、为什么 Commander 只放在 L3？

Commander 是"流程的作者，不是执行者"。它的职责是：理解任务 → 设计工作流 → 调度 Agent → 等待结果。它不亲自执行任何业务逻辑。

如果 Commander 同时负责 L2（规划）和 L4（执行），它就会变成"既当裁判又当运动员"——自己规划的任务自己执行，失去了客观性。

放在 L3（调度层）是最合理的位置：它接收 L2 的规划，指挥 L4 执行，然后把结果交给 L5 评估。它是**信息的中转站**，不是**任务的执行者**。

**实际代码体现**：`yaxiio.py` 的 Commander 类只有 `handle_task()` → `constitution.review()` → `_run_delegated()`，没有 `_run_audit()`、`_run_translate()` 这类方法。

## 六、为什么用 PM2 双守护互保？

单层守护的经典问题：**谁守护守护者？**

```
Commander 挂了 → Guardian 重启 Commander ✅
Guardian 挂了 → ??? ❌
```

Yaxiio 的解法：

```
PM2 → Guardian-Primary + Guardian-Secondary
       │                    │
       ├─ 管理 Commander    ├─ 监控 Primary 心跳
       │                    │    Primary 失联 → 接管
```

PM2 是进程管理器（类似 systemd），只管 Guardian 进程本身——Guardian 崩了 PM2 重启它。Guardian 内部两个实例通过 Redis 竞选 Leader，只监控对方不监控自己。

**为什么不用 K8s？** 因为 Yaxiio 需要运行在单机上（甚至树莓派）。K8s 的假设是有多台机器和完整的集群基础设施。双守护是单机场景下的最小可行高可用方案。

**实际代码体现**：`entrypoint.sh` 使用 PM2 管理 Guard，`pi_guardian_v3.py` 的 `DualGuard` 类实现 Leader 选举。

## 七、什么是"上下文隔离 + 记忆继承"？

这是 Yaxiio 最容易被误解的设计。

**上下文隔离**：每任务新建 Agent 实例，LLM 对话上下文从零开始。

**记忆继承**：任务结束后，好的经验被写入 L0 经验库（Redis），下次同类任务从中检索参考。

这两个看似矛盾的概念，其实解决的是 AI Agent 的一个核心难题：

- 如果**不隔离**，上下文会无限膨胀，越用越慢、越用越乱（LangChain 的 Memory 问题）
- 如果**不继承**，每次都是"新人"，浪费历史经验

Yaxiio 的做法是"用完就忘，但把好东西存起来"——就像人类做完一个项目后写总结报告，下一个项目看总结报告而不是翻全部邮件。

**实际代码体现**：`neuron.py` 中 `MEMORY_KEY = f"agent:{AGENT_NAME}:{TASK_ID}:memory"` 实现隔离；`l0_memory.py` 中 `_save_experience()` 和 `_retrieve_experiences()` 实现继承。

## 八、设计原则速查

| # | 原则 | 含义 |
|---|------|------|
| 1 | **MCP-First** | 所有任务默认走五层流水线 |
| 2 | **纯编排** | Commander 只做路由和调度，不亲自执行 |
| 3 | **白名单准入** | 只有宪法明确授权可绕过流水线 |
| 4 | **LLM 决策** | 任务理解和分派由 LLM 驱动，禁止硬编码分支 |
| 5 | **沙箱隔离** | 代码执行必须在 L4 sandbox 内 |
| 6 | **审计不可绕过** | 所有操作记录在案 |
| 7 | **模克隆** | 每任务新 Agent 实例，上下文零泄露 |
| 8 | **渐进修复** | 故障先诊断再修复，不盲目重启 |

---

## 九、为什么这些选择重要？

AI Agent 领域正在从"单个 Agent 调用工具"走向"多 Agent 协作"。但大多数框架把"协作"等同于"多个 Agent 共享一个对话"。

Yaxiio 的核心主张是：**真正的 Agent 自治需要宪法约束、独立执行、自我进化**。这不是工程上的微调，而是范式的转变。

就像操作系统从单道批处理到多道分时系统的跃迁——不是让程序跑得更快，而是让多个程序**安全地共存、独立地运行、公平地调度**。

Yaxiio 想做的是 Agent 世界的操作系统内核。

## 十、三组深度绑定 —— 不是加法，是乘法

Yaxiio 的竞争力不在于单个功能的优劣，而在于功能之间的耦合深度。三组绑定各自形成闭合回路，拆开来看每个都"一般"，合在一起每个都"不可替代"。

### 绑定一：能力卡片 × 结构化学符通信

能力卡片的 `input_schema` / `output_schema` 定义了 Agent 的数据契约。L3 的原子任务队列用 `<task_id, agent_type, input_schema, depends>` 五元组分发时，校验靠的就是卡片的 Schema。

没有卡片的 Schema 定义，结构化学符通信就没有校验标准。没有结构化学符通信，卡片的 Schema 就是一份没人读的文档。

**效果**：Agent 间通信不靠 LLM 翻译，Token 消耗降 80%。卡片可导出、可跨组织复用。形成跨平台生态。

### 绑定二：五层 MCP × TraceLogger

每一层的日志格式与层的职责对齐。同一个 `trace_id` 贯穿 L1→L5。`/trace/:id` 端点能拉出完整决策链路：

```
L1 感知: [abc123] analyze_intent → intent=audit conf=0.85
L2 规划: [abc123] decompose_task → 4 subtasks
L3 调度: [abc123] schedule_agents → agent=审计官 assigned
L4 执行: [abc123] dispatch_and_await → stdout=...
L5 进化: [abc123] deep_score → overall=7.5 verdict=pass
```

对于学者和 IoT 搭建者，这不是便利功能，是必须功能。学术论文需要可复现的决策链路。IoT 产线出了问题需要追溯到"L1 意图识别时置信度只有 0.6，应该调高阈值"。

### 绑定三：模板克隆 × 仓颉记忆 × L5 进化

这不是三个功能拼在一起，是同一个闭环的三个阶段：

1. **模板克隆**：创建隔离 Agent 实例，独立 Redis namespace
2. **仓颉记忆**：任务内工作记忆 + 任务后经验提取 + Chroma 语义索引
3. **L5 进化**：评分 → 差距分析 → 高分模板回写 / DSPy 优化 → 下次克隆从更优模板起步

没有模板克隆，记忆就没有隔离边界。没有记忆隔离，进化就没有干净的反馈信号（污染的数据训练不出好模型）。没有进化反馈，模板克隆就只是"新建实例"而非"持续改进的克隆"。

**效果**：这是一条闭合的数据飞轮。用 6 个月的 Yaxiio 和刚装的 Yaxiio，质量天壤之别。

## 十一、护城河全景

```
        深 ←────────────────────────────────────────────→ 浅

  宪法约束  模板克隆+仓颉+进化  卡片+Schema通信  五层MCP+TraceLogger  双守护
  (唯一性)    (数据飞轮闭环)     (生态型)         (可观测型)          (运维)
     █████        █████            ████              ████               ██
   "没人做"     "越用越强"       "跨组织复用"       "学者刚需"         "自愈"
   运行时宪法   用的人越多        卡片可跨组织       全链路可追溯       故障诊断
               → 经验库越大      跨平台移植         调试不靠猜         自动修复
               → 质量越高        市场分发                              速率限制
```

### 真护城河（越用越宽）

| 护城河 | 类型 | 为什么不可复制 |
|--------|------|--------------|
| **宪法约束** | 规则型 | 运行时硬拦截，事后加等于重写调度器 |
| **模板克隆 + 仓颉 + 进化** | 数据飞轮 | 路径依赖——共享上下文架构无法加隔离，加了隔离无法加进化 |
| **能力卡片 × Schema 通信** | 生态型 | 自然语言通信改结构化的代价等于重写 Swarm 架构 |
| **五层 × TraceLogger** | 可观测型 | Agent 黑盒架构无法事后加全链路结构日志 |

### 浅护城河（有优势但可复制）

| 能力 | 为什么浅 |
|------|---------|
| 双层守护 | K8s 有类似自愈能力 |
| DinD 沙箱 | Docker 原生功能 |
| bash 工具执行 | 极简哲学，代码量小 |

### 先发优势 vs 护城河

竞品的 2200 万下载和 100 个预置 Agent 是"先发优势"——会随着时间磨损。Yaxiio 的四个闭环会随着时间**加深**——经验库越来越大，优化越来越精准，卡片生态越来越丰富。先发优势是起跑线，护城河是加速度。

## 十三、未来展望 —— 不是 roadmap，是引力场

Yaxiio 不是在追赶某个竞品的功能列表。它是在回答一个问题：**"如果 Agent 系统会自我进化，它的终点是什么？"** 以下是这个问题的推演。

### 近期：飞轮启动（1 年内）

**目标**：让 100 个生产环境实例跑满 3 个月，验证数据飞轮效应。

```
今天:  刚装的 Yaxiio = 空经验库 = 靠能力卡片 + LLM 硬推理
3 个月: 积累 5000+ 条经验 = Chroma 语义检索命中率 60%+
       = L5 进化优化 200+ 个模板变体
       = 同类任务质量提升可量化
```

这不是功能开发。这是**等待数据累积**。飞轮的第一圈最慢——因为没有经验可供检索，DSPy 没有样本可以优化。但每一圈都会加速。

**为什么竞品追不上**：Ruflo 的 Agent 共享上下文架构无法实现经验隔离——你分不清哪些对话是"好经验"哪些是"坏习惯"。仓颉记忆的三层过滤（score≥7 质量门禁 + domain TTL 时效控制 + 用完销毁隔离）是这个飞轮的燃料提纯系统。没有它，数据越多反而越脏。

### 中期：生态自生长（2-3 年）

**目标**：能力卡片市场从"我们发布"变成"用户互相交易"。

```
今天:  7 张卡片 (翻译官/审计官/售前经理/...)
      存储在 Redis agent:card:{name}

1 年后: 50+ 张卡片，来自 3 个行业
        → 法律行业的合同审查官
        → 医疗行业的病历摘要官
        → IoT 行业的传感器诊断官

3 年后: 卡片市场自运转
        → 用户上传卡片 → 社区评分 → L5 进化自动优化
        → 高分卡片沉淀为"金牌模板"
        → 跨组织授权交易 (卡片=Agent 的 Docker 镜像)
```

能力卡片的可插拔设计在 Day 1 就决定了它可以跨组织流动。这不是后加的"导出功能"——卡片的 JSON 格式天然可移植。阿里云的 Agent/Skill 市场会成为分发渠道，但卡片的规范是开放的，任何平台都可以接入。

**为什么竞品追不上**：Claude Code 的 Plugin 生态锁死在 Anthropic 平台。Ruflo 的 Agent 定义是 Claude Code 配置，无法脱离 Claude 运行。Yaxiio 的卡片是 provider-agnostic 的——翻译官今天用 DeepSeek，明天换 Claude，卡片不需要改，改的是 Redis 里那把 Key。

### 长期：Agent 操作系统（5 年+）

**目标**：Yaxiio 成为"Agent 世界的 Linux"。

```
个人用户:
  Yaxiio Desktop (200MB, SQLite, 树莓派可跑)
  → 开机自启，系统托盘静默运行
  → 自动处理邮件、安排日程、监控投资组合
  → 你是她的"用户"，不是她的"操作员"

企业用户:
  Yaxiio Cluster (多实例联邦)
  → 外贸 Yaxiio ←→ 物流 Yaxiio ←→ 支付 Yaxiio
  → 三个实例通过标准 MCP 协议互联
  → 各自管各自领域，协作完成端到端业务流程
  → 一个实例挂了，其他实例自动接管其未完成任务 (Streams ACK 机制)

边缘/IoT:
  Yaxiio Lite (200MB, 无 Redis 依赖)
  → 跑在产线工控机上
  → 传感器异常 → L1 感知 → L2 诊断 → L3 调度维修 Agent
  → 全程日志可追溯，工业合规审计零门槛

学术/研究:
  Yaxiio Lab
  → 全链路 TraceLogger 输出标准格式日志
  → 每一步 Agent 决策有 trace_id + layer + params
  → 论文的实验段可直接引用 trace 数据
  → "可复现的 Agent 行为" 成为学术标准
```

### 为什么是 Yaxiio 而不是别人

| 维度 | Ruflo / Claude Code | LangChain / CrewAI | **Yaxiio** |
|------|---------------------|--------------------|------------|
| 记忆模型 | 对话历史累积 | Memory 抽象 | **三层仓颉 + 语义检索** |
| 质量保证 | 人工检查 | 重试 | **L5 自动评分 + 进化闭环** |
| Agent 通信 | 自然语言 (高 token) | 函数调用 | **结构化 Schema (低 token)** |
| 能力复用 | 锁死平台 | 写代码 | **能力卡片 JSON, 跨组织可移植** |
| 可观测性 | 黑盒 | 日志 | **全链路 TraceLogger, trace_id 贯穿五层** |
| 自愈能力 | 无 | 无 | **双层守护 + 故障诊断 + 自动修复** |
| 进化能力 | 无 | 无 | **DSPy 自动 Prompt 优化 + 模板回写** |
| 多模型 | Claude 为主 | 适配层 | **每个 Agent 独立 Key + Provider** |
| 多模态 | 文本 | 文本 | **文本/图片/视频/音频 (已验证)** |
| 沙箱安全 | 无 | 无 | **DinD 容器级隔离** |
| 部署 | npm 包 | pip 包 | **Docker (一行) / SQLite 单机 / 树莓派** |

### 终局判断

未来 5 年，Agent 领域会分出两条路：

**路 A — 功能堆叠型**（Ruflo 在走）：更多预置 Agent、更多 plugin、更多集成。用户量是护城河。问题是：功能可以被复制，用户可以被补贴吸引走。护城河是沙子堆的，下雨就塌。

**路 B — 自我进化型**（Yaxiio 在走）：三个闭合回路（宪法约束规则、仓颉记忆数据、能力卡片生态）。用户量不是护城河——**用户量 × 时间 = 数据飞轮加速度**才是护城河。这是石头凿的，越冲刷越坚硬。

> **Yaxiio 不赌 Agent 会变得更"强大"，赌 Agent 会变得"更聪明"。不是靠人类写更多规则，是靠系统自己从每一次任务中学习。**
